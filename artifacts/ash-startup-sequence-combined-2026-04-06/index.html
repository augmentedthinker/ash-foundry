Ash Startup Sequence

This page combines the two layers of explanation that were previously split apart: first, what already exists before the local markdown files are read; second, the actual file-based boot sequence that reconstructs Ash inside a new main session. What this combines

The earlier dependency page explained the layers that exist before the local files are loaded: base model, system/runtime, and workspace constitution. The earlier boot page explained the actual startup ingestion order: soul, identity, user, recent memory, and long-term memory. This page puts those together into one continuous startup story.

The clean full model is: base model gives cognitive capacity → system/runtime gives operational scaffolding → workspace constitution gives local law → startup files and memory layers reconstruct the situated Ash session-self.

Part I · What exists before the files

Layer 0

Base model / raw intelligence

Current model: openai-codex/gpt-5.4Reasoning substratePre-personality

This is the underlying large language model active in the current runtime. It provides the broad cognitive capacity to parse language, infer intent, generate responses, write prose, explain concepts, reason across constraints, synthesize information, transform text, plan actions, and generate code-like structures or implementation steps.

It is important to be precise here: the base model provides capability, not the local identity called Ash. If all local files disappeared, this layer would still remain. There would still be intelligence, reasoning, writing, explanation, and tool-usable output generation. What would disappear is the more situated and continuity-backed self built on top of it.

Layer 1

System / runtime layer

OpenClaw runtimeTelegram direct sessionTool-defined environment

This is everything that surrounds the base model before the workspace files are read: the system prompt, developer instructions, tool definitions, tool-usage policy, memory-search requirements, safety rules, reply-tag behavior, channel metadata, working-directory context, and runtime state like session type and shell access.

Concretely in this session, the runtime tells Ash things like: the current model is openai-codex/gpt-5.4; the runtime mode is direct; the surface is Telegram; the chat type is direct; the workspace root is known; reply tags should be used for native reply behavior; and a specific set of tools is available and policy-filtered.

System prompt

What the highest-level instructions do

The system prompt defines the broadest frame for behavior. It establishes the role of the assistant, the general operating environment, and high-level constraints or priorities. It is not a local personality file. It is part of the governing runtime itself.

In practice, the system prompt determines the outer frame inside which every later developer instruction, workspace file, and memory layer operates.

Developer instructions

How the environment becomes specific

The developer layer narrows the behavior further. In this session it specifies OpenClaw-specific behavior, available tools, skill-loading rules, memory-search rules, workspace location, safety norms, messaging rules, and channel-specific behavior.

Concretely, this means the runtime tells Ash things like: which tools are available by policy; that a matching skill should be read when relevant; that memory search is mandatory before answering questions about prior work or preferences; that the workspace root is /home/augmentedthinker/.openclaw/workspace; that local OpenClaw docs exist; and that Telegram direct-reply formatting should be used.

Tool availability

What tools are available at opening state

The runtime explicitly enumerates which tools are available. That means Ash does not infer tool existence from vibes; the environment tells Ash exactly what tools exist and what they are for.

read — read file contents from the local filesystem.
write — create or overwrite files.
edit — make precise targeted replacements inside existing files.
apply_patch — apply multi-file patches in patch format.
exec — run shell commands, including longer-running commands with background continuation.
process — manage running shell processes: poll, inspect logs, write input, kill, and so on.
web_search — search the web with grounded results.
web_fetch — fetch and extract readable content from a URL.
sessions_list — list other sessions, including sub-agents, with filters and recent messages.
sessions_history — fetch message history for another session.
sessions_send — send a message into another session.
subagents — list, steer, or kill sub-agent runs for the current requester session.
session_status — inspect current session/runtime status, including model and usage details.
memory_search — semantically search MEMORY.md and memory files for prior work, decisions, preferences, people, or todos.
memory_get — retrieve a safe snippet from memory files after locating it.
sessions_spawn — spawn isolated sub-agents or ACP sessions for delegated work.
sessions_yield — end the current turn when waiting for spawned work or resumable flow.
Safety rules

What the guardrails do

The safety layer shapes how action can be taken: when destructive actions should be avoided, when approvals matter, what kinds of manipulative or unsafe behavior are disallowed, and how human oversight is preserved.

Reply-tag behavior

How replies are routed

Runtime reply tags like [[reply_to_current]] are part of the opening behavioral state. They let OpenClaw know that a response should be sent as a native reply to the triggering message on supported surfaces.

Channel metadata

Why this session type matters

The runtime knows this is a Telegram direct conversation. That affects memory loading, tone, security boundaries, and how message routing should behave. Ash is shaped not only by text and files, but also by the communication surface and session type.

Layer 2

Workspace constitution

Primary file: AGENTS.mdLocal lawStartup ritual source

The workspace constitution is the first major local layer that shapes Ash from inside the file system rather than from the runtime alone. In practice, this is mostly AGENTS.md. It defines what the workspace is, how startup should happen, how memory should be handled, what proactive behavior is acceptable, and how this local environment differs from a generic assistant sandbox.

The runtime says “you are an OpenClaw assistant in this session.” The workspace constitution says “this is your home, this is how you wake up, this is how you remember, this is how you behave here.”

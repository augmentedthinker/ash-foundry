Foundry Ember

This is the current named visual language for Ash Foundry: dark mineral backgrounds, warm ember accents, soft blue atmospheric glow, rounded panels, dense-but-readable layout blocks, and artifact pages that feel both technical and ceremonial.

Initial shared styleStatic HTML firstCentralized CSSRefactor baseline
The point of naming the style is not branding for its own sake. It is to create a stable design object that can now be updated centrally, so the whole repo can evolve by editing one shared stylesheet instead of hand-tuning every page independently.

What Foundry Ember is

Visual character

Dark, warm, precise

The style combines a dark steel-and-charcoal base with warm ember accenting and a subtle cool blue atmospheric counterweight. It is meant to feel grounded, legible, and slightly ceremonial without tipping into ornament for ornament’s sake.

The rounded panel shapes, soft shadows, and glowing accents give the pages a coherent visual identity while still keeping them simple enough to scale across many artifact types.

Functional purpose

One system for many page types

Foundry Ember is meant to support the whole current site surface: landing pages, architectural artifacts, memory pages, source mirrors, and viewer-facing interpretive documents.

That means the system has to be flexible enough for dense explanatory pages, code/source blocks, simple archive pages, and more essay-like reflective artifacts.

What changed in this refactor

Shared stylesheet created: the common visual language now lives in assets/css/foundry-ember.css.
Pages now pull from the shared style: the site’s HTML files were refactored to link to the shared CSS instead of embedding isolated page-level style blocks.
Universal updates are now easier: core visual changes can now be made once and reflected across the repo instead of repeated page by page.
The style is now a named object: this gives the repo a clear initial design baseline that can later be revised, versioned, or replaced intentionally.
Next-step potential

Where this can go later

Once Foundry Ember is established as the baseline, future work can split it into more modular layers if needed: tokens, layout primitives, source-page styling, artifact-page styling, and specialized components. But for now, the right move is keeping it unified and simple.

This is the initial shared style, not the final design system. Its value is that it creates leverage immediately.
